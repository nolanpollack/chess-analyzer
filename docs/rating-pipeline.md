# Rating Pipeline

How chess-analyzer converts raw Stockfish evaluations into the Elo estimates
and dimensional ratings shown on the profile page.

---

## 1. Per-move accuracy (move quality score)

Every analyzed move receives an accuracy score (0–100%).

**Formula (Lichess/Chess.com win-probability method):**

```
winPctBefore = 100 / (1 + exp(-0.00368208 × evalBefore))   // from mover's perspective
winPctAfter  = 100 / (1 + exp(-0.00368208 × evalAfter))
winPctLost   = max(0, winPctBefore − winPctAfter)
moveAccuracy = max(0, 103.1668 × exp(-0.04354 × winPctLost) − 3.1669)
```

Source: `src/lib/analysis/accuracy.ts` → `computeMoveAccuracy`

**Classification labels** are derived from `winPctLost` (not centipawns directly):

| Label | Win% Lost |
|---|---|
| Brilliant | ≤2%, sacrifice, non-winning position |
| Great | ≤2%, turns a losing game into a winning one |
| Best | Played the engine's top move |
| Excellent | ≤2% |
| Good | ≤5% |
| Inaccuracy | ≤10% |
| Mistake | ≤20% |
| Blunder | >20% |
| Miss | Opponent just erred; player failed to convert |

Source: `src/lib/move-classification.ts` → `classifyMove`

## 2. Game accuracy (Lichess formula)

Per-game accuracy for each color uses the Lichess volatility-weighted formula,
which weights critical positions (high win% swings) more heavily than quiet ones.

**Steps:**
1. Compute win% for every board position in the game (N moves → N+1 positions).
2. Build a sliding window of size `max(2, min(8, floor(N/10)))` over those positions.
3. Weight each move by `stddev(window)`, clamped to `[0.5, 12]`.
4. Compute per-color accuracy as the mean of:
   - **Volatility-weighted mean**: weighted by position criticality
   - **Harmonic mean**: penalises occasional poor moves more than arithmetic mean
5. Final = `(weightedMean + harmonicMean) / 2`, rounded to 1 decimal.

Source: `src/lib/scoring/game-accuracy.ts` → `computeGameAccuracy`

The result (`accuracy_white`, `accuracy_black`) is stored on the `analysis_jobs` table.

---

## 3. Player overall Elo estimate

Simple mean of per-game Lichess accuracy across the trailing 20 analyzed games.
Each game contributes one data point regardless of length — this avoids the
endgame-length bias that move-count weighting would introduce.

Games without a completed analysis (`accuracy_white/black IS NULL`) are excluded.
`sampleSize` in the return value counts games, not moves.

Source: `src/lib/scoring/overall.ts` → `getPlayerOverall`

---

## 4. Elo mapping

Two functions, two use cases:

**`gameAccuracyToElo(accuracy, timeControlClass)`** — used for per-game scores
and player overall rating. Uses a calibrated linear formula per time control:

```
Elo = slope × accuracy% + offset     (clamped to [400, 3000])
```

Coefficients live in `src/config/rating-formula.ts` and are generated by
`scripts/calibrate-rating-formula.ts`. Until that script is run, placeholder
values (`slope=40, offset=-1200`) are used for all time controls.

**`accuracyToElo(accuracy)`** — used for dimensional/factor ratings, which
aggregate across time controls. Uses the `rapid` formula as a neutral baseline.

Source: `src/lib/elo-estimate.ts`

### Why separate formulas per time control

A 75% accuracy game means different things across time formats. In bullet,
players have seconds per move and lower overall accuracy is expected at every
Elo level. The same accuracy would correspond to a much higher Elo in classical.
Running the calibration script independently per time control captures this shift.

### Calibration script

`scripts/calibrate-rating-formula.ts` fetches Lichess games that already have
engine analysis (`evals=true`), computes game accuracy using our pipeline, then
fits `rating = slope × accuracy + offset` per time control via linear regression.
It writes the result directly to `src/config/rating-formula.ts`.

Seed users come from the Lichess leaderboard top-50 per time control, expanded
one BFS hop to their opponents for broader rating coverage. For coverage below
~1400 Elo, pass known low-rated Lichess usernames as CLI arguments:

```
bun scripts/calibrate-rating-formula.ts someuser900 anotheruser1100
```

## 5. Dimensional ratings (factor breakdown)

For each dimension (phase, piece, agency, concept) and each value within that
dimension (e.g. phase=endgame), the scoring engine:

1. **SQL aggregate**: compute raw accuracy from `move_tags ⋈ moves` filtered to
   moves in the trailing-20-game window that have that dimension tag.
2. **Bayesian shrinkage**: blend raw accuracy toward the player's overall accuracy:
   ```
   adjusted = (n × raw + k × prior) / (n + k)
   ```
   `k = 50` (shrinkage strength), `prior = player overall accuracy`, `n = sample moves`.
3. **Elo mapping**: apply `accuracyToElo(adjusted)` for the displayed rating.

Results are cached in `dimension_score_cache` at `(player_id, dimension_type, window_key)`
granularity. Cache is invalidated player-wide after each successful analysis.

Source: `src/features/ratings/server/queries.ts`  
Shrinkage: `src/lib/scoring/shrinkage.ts` → `applyShrinkage` (only call site)

## 6. Known problems

### 6.1 Mate score inflation (primary cause of inflated Elo estimates)

Stockfish encodes forced mates as ±100000 centipawns (a sentinel, not a real
eval). `cpToWinPct(100000) ≈ 100%` — the sigmoid is saturated.

**Effect**: any move that STAYS within the forced-mate zone (eval 100000→100000)
receives 100% accuracy. Lower-rated games frequently have forced-win positions
that persist for many moves. Every suboptimal move in that zone (e.g. missing
mate-in-1, taking 10 more moves to mate) still scores as perfect.

This inflates game accuracy and therefore the player's Elo estimate. A player
rated 700–1100 can receive a 2269 Elo estimate from this effect alone.

**Unit test confirming the behavior**: `src/lib/analysis/accuracy.test.ts`

**Fix**: clamp evals to `±ANALYSIS_CONFIG.evalClamp` (1500cp) before calling
`cpToWinPct` everywhere in the accuracy pipeline. At 1500cp the sigmoid is
99.6% — still expressing a large advantage, but not saturated. Moves within
forced-win positions will then show real win% deltas.

Note: `evalClamp` is already applied for eval display in the UI. It must also
be applied in `computeMoveAccuracy` and `computeGameAccuracy`.

### 6.2 ~~Player overall uses per-move mean~~ — resolved

`getPlayerOverall` now uses a simple mean of per-game Lichess accuracy from
`analysis_jobs` (selecting `accuracy_white` or `accuracy_black` based on
`games.player_color`). Each game counts once regardless of move count.

### 6.3 Elo formula needs calibration run

The per-time-control formulas exist in `src/config/rating-formula.ts` but are
initialized with placeholder values (`slope=40, offset=-1200`) that have not been
validated against actual player ratings.

**Fix**: run `bun scripts/calibrate-rating-formula.ts` with a broad spread of
seed users (leaderboard + low-rated seeds) to populate real coefficients.
